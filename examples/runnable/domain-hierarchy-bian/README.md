# BIAN Domain Hierarchy Example

> **Deprecated.** This example is incomplete and is not runnable as-is. The BIAN
> source data file (`examples/data/bian-service-landscape.json`) has been removed
> from the repository because the upstream data carries no explicit open-source
> license. See [Source data provenance](#source-data-provenance) below if you
> need to re-derive the file.
>
> For a working taxonomy-hierarchy example using freely-licensed data, see
> [`examples/runnable/domain-hierarchy-fibo/`](../domain-hierarchy-fibo/).

Creates the full [BIAN (Banking Industry Architecture Network)](https://bian.org) Service Landscape as a three-level DataHub domain hierarchy:

```
Business Area (8)
  Business Domain (43)
    Service Domain (326)
```

Total: **377 domains** by default. Use `business_areas_filter` to narrow the scope.

## Prerequisites

- DataHub OSS or DataHub Cloud instance
- Personal access token with **Manage Domains** privilege
- Terraform >= 1.11

## Usage

```bash
export DATAHUB_GMS_URL="https://your-instance.acryl.io/gms"
export DATAHUB_GMS_TOKEN="your-personal-access-token"

terraform init
terraform apply
```

To narrow the scope to one or two business areas:

```bash
terraform apply -var 'business_areas_filter=["customers"]'
terraform apply -var 'business_areas_filter=["customers","products"]'
```

Available BIAN business-area ids: `business-development`, `business-management`, `channels`, `customers`, `finance-and-risk-management`, `operations`, `products`, `resource-management`.

## Verify

Open the Domains page in the DataHub UI at `$DATAHUB_GMS_URL/domains` (strip the `/gms` suffix for UI access) to browse the hierarchy.

## Cleanup

```bash
terraform destroy
```

Terraform's dependency graph destroys service domains before business domains, and business domains before business areas, satisfying DataHub's requirement that child domains are deleted before their parents.

## Notes

- All domain IDs are prefixed with `tf-example-bian-{ba|bd|sd}-` to keep them isolated from any BIAN taxonomy you may have loaded via the DataHub Python SDK.

## Source data provenance

The removed `examples/data/bian-service-landscape.json` was a hand-assembled JSON file (not generated by a script). It combined two sources:

**Hierarchy (3-level structure):** extracted from [`biaohao/bian-for-data-governance`](https://github.com/biaohao/bian-for-data-governance), a community repository containing the BIAN v11 taxonomy as structured data. That repository carries **no explicit open-source license**, which is the reason the file was removed from this repository.

**API spec cross-references:** the `has_api_spec_v13` field was derived by checking which service domain names appeared in the OpenAPI 3.x YAML files published by the official BIAN organisation at [`bian-official/public` release 13.0.0](https://github.com/bian-official/public/tree/main/release13.0.0/semantic-apis/oas3/yamls). Those specs are published by BIAN under a permissive licence.

**Counts:** 8 business areas, 43 business domains, 326 service domains from the v11 hierarchy. 245 service domains had a matching v13 API spec; 81 did not. An additional 5 service domains exist in the v13 specs but do not appear in the v11 hierarchy (listed in the original JSON as `service_domains_v13_unclassified`).

To re-derive the file with a clear licence, the recommended path is to source the hierarchy directly from BIAN's official materials (check [`bian.org`](https://bian.org) or the `bian-official/public` repository for a licenced taxonomy export) and assemble the JSON in the same shape expected by `main.tf`.
