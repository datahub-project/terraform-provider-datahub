# DataHub Metadata Model Characteristics and Terraform Resource Design

This document captures field-derived design principles for this provider, grounded in how DataHub's metadata model actually behaves. It should be read before implementing any new resource or data source.

## URN Determinism by Entity Type

Every entity in DataHub is identified by a URN. URN keys fall into two categories: deterministic (derived from a human-readable name) and non-deterministic (randomly generated, typically when the entity is created via the UI). This distinction is the single most important thing to get right when adding a resource, because non-deterministic URNs make multi-environment consistency impossible.

**The rule:** the provider must always produce a deterministic URN for any resource it manages, and that URN must match the convention used by the DataHub Python SDK and CLI for the same entity type.

The reason for matching the SDK convention specifically is that customers will have existing entities created by the SDK or by ingestion. If the provider uses a different URN convention for the same human-readable name, a Terraform-managed entity and an SDK-created entity with the same name will have different URNs and will coexist as duplicates rather than converging.

### Entity-type URN key strategy

| Entity type | URN format | Key source | Notes |
|---|---|---|---|
| `dataHubIngestionSource` | `urn:li:dataHubIngestionSource:<id>` | User-supplied `source_id` | Already implemented correctly |
| `tag` | `urn:li:tag:<tagName>` | Human-readable name, verbatim | Do NOT use a hash; SDK uses plain name |
| `glossaryTerm` | `urn:li:glossaryTerm:<qualifiedName>` | Dot-separated qualified name | e.g. `classification.PII` |
| `glossaryNode` | `urn:li:glossaryNode:<qualifiedName>` | Dot-separated qualified name | Same pattern as terms |
| `domain` | `urn:li:domain:<uuid>` | User-controlled UUID or hash from name | Must be stable; do not let DataHub auto-assign |
| `corpGroup` | `urn:li:corpGroup:<groupName>` | Group name verbatim | |
| `corpUser` | `urn:li:corpuser:<username>` | Username verbatim | Note: `corpuser` not `corpUser` in the URN |
| `dataHubPolicy` | `urn:li:dataHubPolicy:<uuid>` | User-controlled UUID | Stable across envs; accept as resource input |

### Container IDs: a special case

Container URNs (databases, schemas, catalogs) are generated by each source connector using a connector-specific hashing algorithm applied to a dictionary of path components. The hash algorithm and dictionary structure differ per connector type, making it impossible for the provider to compute or predict container URNs without reimplementing connector-specific logic.

Consequence: the provider should **never** attempt to construct container URNs. Any resource that needs to reference a container (for example, a dataset enrichment resource) should accept the full URN string as a plain input attribute. A future `datahub_container` data source (which looks up the container by platform, environment, and path) would be the right way to resolve container URNs dynamically within a Terraform configuration.

## Reference Validation and Terraform Dependencies

DataHub's metadata model is loosely coupled by design: you can write aspects that reference entities that do not yet exist, and DataHub will accept them. The referenced entity can be created later and the reference will resolve retroactively. Most entity types do not enforce referential integrity at write time. (Structured properties are an exception — they are validated at assignment time.)

**The advantage for Terraform:** when provider resources are connected through Terraform expressions (e.g., `datahub_tag.my_tag.urn`), Terraform's dependency graph ensures that the referenced resource is created before the referencing resource, without any special handling in the provider. This is a genuine advantage over imperative scripts, which must either do topological sorting manually or accept that ordering errors will produce broken references.

**Where this breaks down:** users can bypass Terraform's dependency system by typing URN strings directly into attributes. The provider cannot validate these at plan time without making API calls (which breaks offline plan semantics). Raw URN string inputs should be documented as "the user is responsible for ensuring the referenced entity exists."

**Design implication:** for enrichment resources that reference governance objects (tags, glossary terms, domains), model the input as a resource reference attribute (accepting the `.urn` output of another resource) as the primary pattern, with a raw URN string as a documented escape hatch. Do not make raw strings the only option.

## Upsert Semantics for List-Valued Attributes

DataHub's OpenAPI v3 entity endpoints perform a full upsert of each aspect: the submitted value replaces the entire aspect, including any list-valued fields within it. There is no native "add one item to a list" endpoint in the OpenAPI surface that is also idempotent.

**The Terraform-idiomatic approach:** resources that manage aspects containing lists (tag assignments, owners, terms, etc.) should own the **complete** list as their desired state. A `terraform apply` pushes the full list, overwriting whatever was there before. This is idempotent: applying the same plan twice produces the same result. It also means that items added outside Terraform (via the UI or SDK) will be removed on the next apply — which is the correct behavior for a resource that Terraform fully manages.

**Do not use PATCH/append semantics** for any resource the provider manages. Append is not idempotent (applying twice would duplicate items), and it depends on the prior state of the server in a way that makes plans unreliable.

**Operator guidance:** document clearly in the resource docs that the provider owns the full list for any list attribute. Operators who want to allow business users to add tags/owners interactively should not use Terraform to manage those attributes on those entities.

## Delete Behavior

DataHub distinguishes two deletion modes:

- **Soft delete:** sets `status.removed = true` on the entity. The entity is hidden from normal queries but its data remains in the database. Subsequent ingestion or a provider re-create can reactivate the entity, restoring its previous aspects.
- **Hard delete:** removes the entity from the database entirely.

The OpenAPI v3 entity endpoints use soft delete by default when you issue an HTTP DELETE. Soft-deleted entities can be reactivated by writing any aspect back to the same URN.

**Implication for `terraform destroy`:** after a soft delete, if a user re-creates the same resource (same URN) with `terraform apply`, any aspects that are not included in the new resource schema will retain their previous values from the soft-deleted entity. For ingestion sources this is benign because the provider writes all relevant aspects on create. For future entity types with many optional aspects, this may cause unexpected state.

**The reactivation / soft-delete resurrection bug:** if a resource is soft-deleted and then the same URN is re-ingested by a connector before the provider recreates it, the entity will be reactivated with a mix of old and new aspects. This is a DataHub platform behavior that the provider cannot prevent; document it in resource guides for entity types where this scenario is likely.

**Not all entity types support `status.removed`.** Before implementing delete for a new entity type, verify that the entity type has a status aspect and that the OpenAPI DELETE endpoint behaves as expected. Entity types that lack a status aspect may require a hard delete (a different API call) or may have no supported delete path.

**Provider option:** consider exposing a `force_hard_delete` attribute on resources where clean-slate semantics matter, so operators can opt into purging the entity from the database on destroy.

## Provider Scope

Field experience across multiple customer implementations converges on a clear division of labor:

**In scope for Terraform (platform-level configuration, engineering-team-owned):**
- Ingestion source configuration and recipes
- Foundational governance taxonomy: tags, glossary nodes and terms, domains
- Access policies and RBAC configuration
- Structured properties (schema)

**Out of scope or inadvisable for Terraform (enrichment, business-user-owned):**
- Descriptions and documentation on individual datasets, fields, or other data assets
- Tag assignments and glossary term assignments on individual datasets
- Ownership assignments on individual data assets
- Any metadata that business users are expected to create or edit interactively via the DataHub UI

The reason for this boundary: DataHub's primary value proposition as a collaborative data catalog depends on business users being able to enrich metadata through the UI. If Terraform manages the same enrichment data, every `terraform apply` will overwrite manual edits. This creates operational friction and erodes trust in the catalog. The provider should be positioned as the right tool for platform engineers managing platform configuration, not for analysts managing dataset-level metadata.

**Corollary:** the provider should not attempt to manage data asset entities (`dataset`, `dataJob`, `dataFlow`, `chart`, `dashboard`) as Terraform resources. Those entities are the output of ingestion, not inputs to it. Enrichment aspects on those entities (descriptions, ownership, tags) are in the out-of-scope category above. The correct Terraform primitives for data assets are **data sources** (for looking up URNs and metadata), not resources.

## Cross-Environment Parameterization

The standard use case for this provider is managing the same configuration across multiple DataHub environments (dev, staging, prod). URN references that are environment-specific — such as executor IDs, recipe hostnames, or secret names — should be expressed as Terraform input variables, not hardcoded in resource definitions.

The provider does not need any special cross-environment logic (such as URN rewriting or environment-aware string substitution). Terraform's variable system handles this: a `tfvars` file per environment, or a workspace-per-environment approach, lets the same resource definitions produce correct URNs for each target.

**Practical pattern for recipes:** recipe content often contains environment-specific values (database hostnames, secret references). These should use DataHub's own `${DATAHUB_SECRET_<NAME>}` substitution syntax in the recipe string, combined with separate `datahub_secret` resources (when that resource is implemented) or pre-existing secrets managed outside Terraform. This keeps credentials out of Terraform state and out of the recipe attribute value.

## The Terraform Provider as a Control Plane

DataHub provides both a Terraform provider (this project) and an MCP server for AI agent integration. These are complementary, not competing:

- The MCP server is the right interface for interactive and exploratory agent use: discovering metadata, adding a description to a specific dataset, answering questions about lineage.
- The Terraform provider is the right interface for controlled, auditable, multi-environment changes to platform configuration: defining a new ingestion source, publishing a new tag to the taxonomy, applying a policy.

An emerging pattern is to use Terraform as the control plane for AI-driven changes: rather than an agent writing directly to DataHub via the MCP server, the agent proposes a Terraform configuration change (a pull request editing `.tf` files), which goes through a standard review process before being applied. This preserves the review gate and the audit trail, and it is a natural fit for the entity types already in scope for this provider.
