# Assertion resource modeling: typed vs generic vs codegen

Status: design note / decision record. Written after a brownfield migration field test ([extract-import-field-test.md](extract-import-field-test.md)) exposed how much of DataHub's assertion surface the current typed resources do not cover. Coverage evidence: [assertion-coverage.md](assertion-coverage.md).

## The problem

DataHub's `assertion` is a single entity whose `assertionInfo` aspect is a **tagged union over a large, Cloud-evolving set of types** (FRESHNESS, VOLUME, SQL, FIELD, DATASET, DATA_SCHEMA, CUSTOM, ...), and each type has sub-variants the platform keeps extending. The provider models a few of these as **strongly-typed resources** with firm schemas: `datahub_freshness_assertion`, `datahub_volume_assertion`, `datahub_sql_assertion`, plus `datahub_custom_assertion`.

Live introspection (see the coverage doc) shows those typed resources express roughly the absolute-threshold common case and omit, across the three monitor types: the `*_CHANGE`/growth sub-types, AI-inferred thresholds (`inferWithAI`/`inferenceSettings`), row `filter`s, `failureSeverityConfig`, `backfillConfig`, and `SINCE_THE_LAST_CHECK`. For `datahub_volume_assertion` that is roughly half of the volume-assertion UI.

This is structural: **a typed wrapper over an extensible union always lags the union.** Every new operator/sub-type/parameter DataHub ships requires a provider schema change and release. Continuing to hand-add typed assertion resources commits us to that treadmill indefinitely.

## Scope first: NATIVE assertions only

Before any typed-vs-generic question, the migration scope must be bounded by `assertionInfo.source.type` (`NATIVE` / `EXTERNAL` / `INFERRED`):

- **`NATIVE`** -- authored as a managed monitor. The only Terraform migration target.
- **`EXTERNAL`** -- reported by an ingestion source (dbt tests, Great Expectations). Owned/regenerated by ingestion.
- **`INFERRED`** -- AI/smart-tuned thresholds (the system owns the values).

`EXTERNAL`/`INFERRED` assertions are **out of scope**, the same boundary as ingested lineage and profiles: not author-as-code, regenerated out-of-band, and importing them would both fight the producing system and fan out hundreds of unmanaged resources. The field test made this concrete -- 96 of 101 assertions were `EXTERNAL` (ingested dbt tests); only 5 were `NATIVE`. This also refines the proposed migration rule "all existing assertions must be extractable/importable": it applies to **NATIVE** assertions; EXTERNAL/INFERRED are extractable-as-visible but not import-as-managed.

**Consequence:** enumeration and import filter to `source.type == NATIVE` first. This dramatically shrinks the problem (the scary "7-type union, 96 unmapped" was inflated by ingested dbt tests) and is more principled than filtering by sub-shape. The typed-vs-generic question below therefore concerns only the bounded set of NATIVE assertions.

### AI-tuned assertions are INFERRED, not NATIVE (verified)

Creating a volume assertion with `inferWithAI: true` -- even as a normal user via the API -- makes DataHub stamp it `source = INFERRED`, not NATIVE (verified live, 2026-06-15). There is no "NATIVE assertion with AI-tuned params" hybrid: enabling AI tips the whole assertion to INFERRED. It stays a **singleton** (one assertion + one monitor; no fan-out). The author-set config (`adjustmentSettings`: training lookback, sensitivity, algorithm, exclusion windows) lives on the **monitor** (`assertionMonitor.settings.adjustmentSettings`); the thresholds are **computed** and overwritten by the AI over time, so they must be modelled `Computed`/read-only, never managed as static input.

Two consequences:

- The `source==NATIVE` boundary already handles AI correctly -- AI assertions are INFERRED, so they are excluded from enumeration and refused on import. There is **no drift bug** from importing an AI assertion into a typed resource (an earlier worry); the source filter pre-empts it. And the typed resources should **not** model `infer_with_ai`, since AI assertions are not NATIVE.
- `created.actor` distinguishes a **user-declared** AI monitor (a real corpuser ran `inferWithAI`) from a **system-auto** one (`urn:li:corpuser:__datahub_system`). That opens a *candidate future capability* (see Recommendation): manage user-declared AI monitors as a resource -- adjustment settings as input, thresholds `Computed` -- while system-auto INFERRED stays out of scope. The fan-out concern is a separate entity, `AssertionAssignmentRule` (a parent rule + filter that auto-assigns assertions); a future `datahub_assertion_assignment_rule` could manage the parent while its generated children stay out of scope.

## Three options

### A. Hand-typed resources (current)
- **Pros:** best UX -- typed attributes, enum validation, plan-time errors, self-documenting HCL, first-class references (`entity_urn`).
- **Cons:** chases every enum forever; perpetually behind; one shared entity type fanning out to four resources also breaks the extract registry's one-entity-one-resource model, so monitor assertions cannot be auto-enumerated (they import by URN only).

### B. Generic `datahub_assertion` (raw structured JSON)
A resource taking a JSON `definition` mirroring the API, stored and round-tripped via `jsontypes.Normalized` -- the `aws_iam_policy` / `datahub_ingestion_source.recipe` pattern.
- **Pros:** covers every type and future addition with no provider change; enumeration becomes trivial (one entity, one resource, one enumerator); escapes the treadmill.
- **Cons / why assertions are harder than recipes:**
  1. **Write API is type-segmented.** Unlike ingestion sources' single `createIngestionSource`, assertions are written via separate typed mutations (`upsertDataset{Volume,Freshness,Sql}AssertionMonitor`, `upsertCustomAssertion`). There is no single generic upsert, and writing raw aspects via OpenAPI v3 bypasses the monitor service layer (forbidden -- see the read/write rule in CLAUDE.md). So a generic resource still needs a small per-type **write-dispatch** layer.
  2. **Input shape != stored/read shape.** One write input fans out into two entities and three aspects on read (`assertionInfo` + `assertionActions` on the assertion, `monitorInfo` on a separate Monitor entity -- the split that forced `GetAssertionMonitor`). So `Read` must **assemble and re-nest** an input-shaped document from multiple sources, not pass a blob through. (Worked example in the normalization section below.)
  3. **AI inference breaks round-trip.** With `inferWithAI = true`, the server computes the thresholds and stores concrete numbers the user never wrote, so the read document diverges from config unless the resource tracks "inferred" as separate intent state.
- **Net:** still the best coverage answer, but it is "thin write-dispatch + a read-normalization mapper + an inference carve-out", not a dumb blob.

### C. Schema-driven codegen
Introspect DataHub's GraphQL schema in CI and generate the typed inputs/resources.
- **Pros:** keeps typed UX while making "keep up" mechanical (regenerate + release).
- **Cons:** upfront harness; must run against a **dumped SDL**, not live per-plan, because of the `BadFaithIntrospection` guard; codegen produces the input mapping but cannot infer the read-normalization (how the service splits an input into aspects) -- that stays hand-written.

## Normalization sketch (why B is not a pass-through)

For a demo volume assertion, the generic resource's `Read` reconciles:

```
definition (write input)                     <= read source
type: "ROW_COUNT_TOTAL"                       <= assertion / assertionInfo.volumeAssertion.type
rowCountTotal { operator, parameters }        <= assertion / assertionInfo.volumeAssertion.rowCountTotal
entityUrn                                     <= assertion / assertionInfo.entityUrn
actions                                       <= assertion / assertionActions aspect
evaluationSchedule { cron, timezone }         <= MONITOR / monitorInfo.assertionMonitor.assertions[*].schedule
evaluationParameters { sourceType }           <= MONITOR / monitorInfo...assertions[*].parameters.datasetVolumeParameters
mode                                          <= MONITOR / monitorInfo.status.mode
```

Steps: fetch two entities (assertion + its Monitor via the `Evaluates` link), merge three aspects, re-key/re-nest into the write-input vocabulary, then canonicalize (drop server defaults/nulls) so the assembled JSON semantically equals the user's config and `jsontypes.Normalized` suppresses formatting noise.

## Recommendation

**Revised** after full `__schema` discovery and the AI/INFERRED finding. Those two facts removed the original arguments against extending the typed resources: the NATIVE surface is now fully enumerated (no longer "unbounded/unknown"), and the single hardest, most-treadmill-y piece -- AI-tuned thresholds -- is `source == INFERRED`, hence out of the typed-NATIVE scope entirely (we never have to model `inferWithAI`). What remains for the existing types is static, declarative, fully-specified config -- exactly what IaC is for.

In priority order:

0. **Scope by source first.** Enumeration and import filter to `source.type == NATIVE`. Highest-value, lowest-risk, independent of everything below. **Done -- PR #58.**

1. **Complete the existing typed resources to full NATIVE coverage of their type.** The remaining gaps are bounded, discovered, static, and high-value (clearly author-as-code): volume `ROW_COUNT_CHANGE`, sql `METRIC_CHANGE`, freshness `SINCE_THE_LAST_CHECK`, plus the cross-cutting NATIVE inputs (`filter`, `failureSeverityConfig`, exclusion windows, `backfillConfig`). Extending the existing resources is **incremental** -- the hard part, the two-entity assertion+monitor read, already shipped (#56) -- and completing volume makes *all* NATIVE volume assertions land in one resource, satisfying "all volume into one consistent target" **without** a generic. Start with volume `ROW_COUNT_CHANGE` (see [volume-row-count-change-build.md](volume-row-count-change-build.md)). **Largely done** -- the three sub-types and the `description` / `filter` / `failure_severity` cross-cutting fields shipped; `backfillConfig` and the `failureSeverityConfig.rules` engine are deferred (imperative / niche). See [typed-assertion-completion-plan.md](typed-assertion-completion-plan.md) for the per-increment record and deferral rationale.

2. **Gate NEW typed resources on evidence of NATIVE use.** `datahub_field_assertion`, `datahub_schema_assertion`, and `DATA_JOB_RUN` freshness are not "complete the existing types" -- they are new surface. Column/schema checks in the wild are frequently EXTERNAL (dbt) or INFERRED (smart), so a NATIVE FIELD/SCHEMA resource may serve very few real assertions (the field test had zero FIELD assertions). Confirm demand before building.

3. **Generic `datahub_assertion` demotes to an optional backstop.** Once the existing types are complete, a generic raw-JSON resource is future-proofing -- a catch-all for shapes we deliberately don't type and for future Cloud additions -- not the primary coverage mechanism. Build only if a real need for an unmodeled NATIVE shape appears. (Its read-normalization complexity, sketched above, is another reason to prefer completing the typed resources for the known cases.)

4. **Codegen (C)** -- unchanged: revisit only if assertions become strategically central enough to justify the harness.

This **supersedes** this note's earlier stance ("keep typed minimal, add a generic for the tail, don't extend"). The treadmill caution now applies narrowly: don't add *new* typed resources speculatively (gate FIELD/SCHEMA on demand, item 2), and don't try to model INFERRED/AI in the typed resources (it's out of scope). Completing the *existing* types is the opposite of chasing the union -- it's closing a finite, known set.

Note on enforcement: the `source==NATIVE` scoping (0) shipped in PR #58 on both paths -- extract (source-aware, type-routed enumerators) and import (a point-of-import guard), with acceptance tests. The enumerators currently filter to the sub-shapes each resource models; completing item 1 widens those filters as the resources gain coverage (e.g. volume enumeration expands to include `ROW_COUNT_CHANGE` once the resource supports it).

Candidate future capabilities (not decided, not built; recorded so the option is not lost):

- **User-declared AI monitors.** `inferWithAI` assertions are `source == INFERRED` (see "AI-tuned assertions" above), so they are currently out of scope. But a *user-declared* AI monitor (distinguished from system-auto ones by `created.actor != urn:li:corpuser:__datahub_system`) is a legitimate thing to manage: a resource could declare the intent + author-set adjustment settings (lookback, sensitivity, algorithm, exclusion windows) as inputs, with the AI-computed thresholds modelled `Computed`. This would relax "INFERRED is out of scope" to "system-auto INFERRED is out of scope; user-declared AI monitors are manageable."
- **Assertion assignment rules.** Bulk fan-out is the separate `AssertionAssignmentRule` entity (a parent rule + dataset `filter` that auto-assigns assertions). A `datahub_assertion_assignment_rule` resource could manage the parent rule; its generated child assertions stay out of scope (excluded by source).
