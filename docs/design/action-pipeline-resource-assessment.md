# Design assessment: `datahub_action_pipeline` resource

Status: **BUILT** (was: assessment / recommendation). Written during the datahub-gcp-demo migration field test, where 5 action pipelines (Dataplex + BigQuery metadata sync-back) are the one class of Terraform-managed DataHub object with no provider resource.

**Gate verified (the crux in "Design implications" below): `upsertActionPipeline(urn, input)` DOES create at a caller-chosen, not-yet-existing URN** -- confirmed live: the entity persists with the chosen `dataHubActionKey.value.id` and the recipe round-trips verbatim (including `${SECRET}` placeholders). So `datahub_action_pipeline` was built as a deterministic-URN, Cloud-only, recipe-as-`jsontypes.Normalized`, OpenAPI-read resource (plus a `datahub_action_pipelines` data source). One nuance found: the upsert writes the definition and then reloads/starts the runtime; the reload can return HTTP 500 for a recipe it cannot run while the definition is already persisted, so the resource verifies via the OpenAPI read on a non-cloud-only error and treats a persisted definition as success-with-warning (definition-only scope; run state is not modeled). Validated end-to-end: live create/update/import/destroy, plus an extract/import round-trip of the demo's 11 live pipelines that plans `0 to change`.

## What an action pipeline is

A DataHub Cloud "action pipeline" (a.k.a. automation) runs a packaged action - e.g. propagating glossary terms / descriptions / tags back to Dataplex or BigQuery. In the demo today they are managed with the generic `sullivtr/graphql` provider via `createActionPipeline` / `upsertActionPipeline` / `deleteActionPipeline`, with the recipe carrying `${SECRET_NAME}` placeholders (see `~/src/datahub-gcp-demo/terraform/automations.tf`).

## Feasibility findings (verified against demo.gcp.acryl.io)

| Concern | Finding |
|---|---|
| Entity type / URN | `dataHubAction`; URN `urn:li:dataHubAction:<uuid>`. **UUID is server-generated by `createActionPipeline` - non-deterministic.** |
| Read path (project rule: OpenAPI v3, MySQL-consistent) | `GET /openapi/v3/entity/datahubaction/{urn}` returns HTTP 200 with aspects `dataHubActionKey`, `dataHubActionInfo`, `dataHubActionStatus`. ✅ |
| Readable config | `dataHubActionInfo.value` = `{ name, description, type, category, config { recipe, executorId } }` (+ `debugMode` on write). Recipe is a JSON document. |
| Mutations | `createActionPipeline(input)` (UUID assigned), `upsertActionPipeline(urn, input)` (targets a specific URN), `deleteActionPipeline(urn)`. |
| List (for data source / enumeration) | GraphQL `listActionPipelines` works (demo has 11; the project created ~5, the rest are from elsewhere - same shared-instance caveat as everything else). |
| OSS vs Cloud | Cloud-only: `dataHubAction` and these mutations are absent from OSS GraphQL. Must use `cloudOnlyBadge` + the Cloud-only table, like the assertion monitors. |
| Stability | `docs/roadmap.md` flags action pipelines as experimental ("Newer than action workflows; relationship unclear. Defer.") and the lifecycle mutations sit near the internal/operational deny-list. Treat the API as Cloud-internal: no external stability guarantee. |

## Design implications

1. **Deterministic URN is the crux.** The provider's model (see `docs/design/datahub-model-and-resource-design.md`) requires a deterministic URN so re-creation and SDK coexistence are stable. `createActionPipeline` mints a random UUID, which violates this. `upsertActionPipeline(urn, input)` accepts an explicit URN, so the resource should **generate a deterministic id client-side** (e.g. a hash/slug of `name`, matching how `datahub_ingestion_source` derives `source_id`) and always call `upsertActionPipeline` with `urn:li:dataHubAction:<deterministic-id>` - never `createActionPipeline`. **To verify before building:** that `upsertActionPipeline` will create (not just update) an entity at a caller-chosen, not-yet-existing URN. If it rejects unknown URNs, deterministic URNs are not achievable and the resource should not be built as a standard managed resource.
2. **Recipe as `jsontypes.Normalized`.** The config recipe is JSON, exactly like `datahub_ingestion_source.recipe`; reuse the same custom type (see F-A.6 in `extract-import-field-test.md`) to avoid whitespace/key-order drift.
3. **Secrets.** Recipes reference `${SECRET_NAME}`; document the same secret-handling guidance as ingestion sources (do not inline credentials).
4. **Import + enumeration.** Read/ImportState via the OpenAPI v3 path above. Add a `dataHubAction` import target (CUSTOM-style: enumerate via `listActionPipelines`) to the shared `internal/provider/importtarget/targets` registry so the CLI picks it up automatically. Apply the shared-instance/selective-extraction guidance.
5. **`type` is a free-form action class string** (e.g. `dataplex_metadata_sync`, `datahub_integrations.propagation.bigquery.tag_propagator.BigqueryTagPropagatorAction`). Accept it as a plain required string; do not enum-constrain it (the set is open and Cloud-version-dependent).

## Recommendation

**Buildable, but gate it.** The read path is clean and consistent and the mutation surface is well-defined, so a `datahub_action_pipeline` resource is technically sound and has a real consumer (this demo). However it is Cloud-only, experimental, and API-unstable by the project's own roadmap. Recommendation:

- Proceed only after verifying point (1) (upsert-creates-at-chosen-URN). If that holds, build it behind the Cloud-only badge, deterministic-URN, recipe-as-`jsontypes.Normalized`, OpenAPI-read pattern, as its own PR with a design-doc link and an explicit "experimental / Cloud-internal API" note in the resource description.
- If (1) fails, do not ship a managed resource with non-deterministic URNs; instead expose a read-only `datahub_action_pipeline` data source (lookup by URN) and leave creation to the existing tooling, revisiting when the API stabilizes.
- Until then, the demo's 5 action pipelines stay on the `sullivtr/graphql` approach (documented as the one remaining non-native object in the migration).
