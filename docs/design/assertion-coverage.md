# Assertion resource coverage vs the DataHub API

What each `datahub_*_assertion` resource can express today, measured against the live DataHub Cloud GraphQL schema.

**Method:** full GraphQL schema introspection (`__schema`, 1515 types) against a DataHub Cloud instance, June 2026, cross-checked with live `searchAcrossEntities` results. Note: a `BadFaithIntrospection` guard rejects queries with many repeated `__type` aliases, but the single canonical `__schema` introspection works -- so a complete offline schema dump is one query away.

Legend: yes = expressible; **no** = not expressible by the resource; passthrough = the resource accepts a free-form string so the value is usable even though only a subset is documented.

## Scope: migration targets NATIVE assertions only

DataHub stamps every assertion with `assertionInfo.source.type`, one of:

- **`NATIVE`** -- authored as a managed monitor (UI, API, or this provider). **These are the migration targets.**
- **`EXTERNAL`** -- reported into DataHub by an ingestion source (e.g. dbt tests, Great Expectations). Owned and regenerated by the ingestion pipeline.
- **`INFERRED`** -- AI/smart-tuned thresholds (the system owns the values).

`EXTERNAL` and `INFERRED` assertions are **out of scope for Terraform**, the same way ingested lineage and dataset profiles are: they are not author-as-code platform config, they are regenerated out-of-band, and managing them would fight the system that produces them (and fan out hundreds of unmanaged resources). On the field-test instance this was stark -- 96 of 101 assertions were `EXTERNAL` (ingested dbt tests); only 5 were `NATIVE` (the ones actually authored in Terraform).

Note (verified live): setting `inferWithAI: true` on any of the monitor inputs below makes DataHub classify the assertion `source = INFERRED`. So the `inferWithAI`/`inferenceSettings` rows marked "no" below are not a coverage gap to close in the typed resources -- an AI-tuned assertion is INFERRED and out of scope by the source rule (its thresholds are AI-computed, so they could only ever be `Computed`, never managed input). `created.actor` distinguishes user-declared AI monitors from system-auto ones -- see the candidate future capability in [assertion-resource-modeling.md](assertion-resource-modeling.md).

**Rule:** assertion enumeration / import must filter to `source.type == NATIVE`. Everything below concerns NATIVE assertions.

## The assertion-type universe (what the provider does and does not model)

`AssertionType` has **7** members. The provider implements resources for **4**:

| Assertion type | Upsert input exists | Provider resource |
|---|---|---|
| VOLUME | `UpsertDatasetVolumeAssertionMonitorInput` | `datahub_volume_assertion` (subset -- below) |
| FRESHNESS | `UpsertDatasetFreshnessAssertionMonitorInput` | `datahub_freshness_assertion` (subset -- below) |
| SQL | `UpsertDatasetSqlAssertionMonitorInput` | `datahub_sql_assertion` (subset -- below) |
| CUSTOM | `UpsertCustomAssertionInput` | `datahub_custom_assertion` |
| **FIELD** | `UpsertDatasetFieldAssertionMonitorInput` | **none** (FIELD_VALUES / FIELD_METRIC unmodeled) |
| **DATA_SCHEMA** | `UpsertDatasetSchemaAssertionMonitorInput` | **none** |
| **DATASET** | (native dataset assertion) | **none** (this is the type the 96 EXTERNAL dbt tests use) |

So even for NATIVE assertions the provider covers 4 of 7 types, and within those 4 only a sub-shape each (next sections). DataHub also exposes dimensions none of the resources touch -- `FreshnessAssertionType` (DATASET_CHANGE / DATA_JOB_RUN), exclusion windows, monitor sensitivity, failure-severity rules, AI adjustment settings, and an `AssertionAssignmentRule` subsystem for bulk-assigning assertions across many datasets.

| Source type (`DatasetVolumeSourceType`): `INFORMATION_SCHEMA, QUERY, TABLE_STATISTICS, DATAHUB_DATASET_PROFILE, PLATFORM_API` | 5 | passthrough (3 documented) |

## Status after the typed-completion build (see typed-assertion-completion-plan.md)

The `*_CHANGE` sub-types (volume `ROW_COUNT_CHANGE`, sql `METRIC_CHANGE`), the `SINCE_THE_LAST_CHECK` freshness schedule, and the cross-cutting `description` / `filter` (volume+freshness) / `failureSeverityConfig.defaultSeverity` (freshness+sql) fields are now modeled. The field-availability per input was corrected against the live schema during that build: `failureSeverityConfig` is on the **freshness and sql** inputs only (not volume), and `backfillConfig` is on the **volume** input only.

Remaining gaps, all deliberate:

- `inferWithAI` + `inferenceSettings` -- AI/smart thresholds. Setting `inferWithAI` makes DataHub classify the assertion `source = INFERRED`, which is out of scope by the source rule. Never modeled.
- `failureSeverityConfig.rules` -- the conditional per-result severity escalation list. `defaultSeverity` is modeled; the rules engine is deferred (list-ownership semantics, niche).
- `backfillConfig` (volume) -- a one-shot imperative bootstrap: it seeds historical evaluations from a start date, is constrained to a 365-day lookback, and does **not** round-trip into the assertion or monitor entity on read. Modeling it as a managed attribute would produce perpetual drift, so it is deferred as out-of-band imperative work rather than IaC state.
- Operators: 6 of 18 documented (passthrough -- the others are usable, just undocumented).
- `MonitorMode` is `ACTIVE, INACTIVE, PASSIVE`; resources document `ACTIVE`/`PASSIVE` (passthrough, so `INACTIVE` is usable).

## Interpretation

Two layers of scoping apply, and they pull in opposite directions:

- **Scope first by source.** Filtering to `source.type == NATIVE` removes most of the apparent gap: the `EXTERNAL`/`INFERRED` assertions (ingested dbt tests, smart/AI monitors) are out of scope, so the unmodeled `DATASET`/`FIELD` types and the bulk-assignment subsystem largely fall away as migration concerns -- they are ingestion/auto artifacts, not author-as-code. This is the single most important framing: the assertion-migration surface is NATIVE assertions, which on the field-test instance was 5, not 101.

- **Within NATIVE, the typed resources still cover only a slice.** Of the NATIVE monitor types, the provider models 4 of 7 assertion types, and within those only the absolute-threshold common case -- missing the change/growth sub-types, AI-inferred thresholds, filters, severity, backfill, and the FIELD/SCHEMA types entirely. A platform team authoring NATIVE assertions in code can hit these limits.

This matrix is the evidence base for the typed-vs-generic-vs-codegen decision in [assertion-resource-modeling.md](assertion-resource-modeling.md). The headline conclusions: (1) migration enumeration/import filters to `source.type == NATIVE`; (2) the residual typed-coverage gap, now bounded to NATIVE assertions, is what the generic `datahub_assertion` escape hatch exists to close.
